[CLAUDE SELF-DIRECTED OPERATING MANUAL — 해설서]
부제: 이 운영 문서를 Claude에게 주면 무엇이 달라지고,
어떻게 설치하고 사용해야 하는가


======================================================================
0. 이 해설서의 목적
======================================================================

함께 제공된 `CLAUDE_SELF_DIRECTED_OPERATING_MANUAL.txt`는
Claude에게 “이 기능을 만들어라”라고 지시하는 단발성 프롬프트가 아니다.

그 문서는 Claude가 프로젝트에서 일하는 방식을 정의한다.

쉽게 말하면:

- PRODUCT_SPEC는 무엇을 만들지 설명한다.
- METHODOLOGY는 음악적으로 어떤 원리를 사용할지 설명한다.
- IMPLEMENTATION_PLAN은 어떤 순서로 만들지 설명한다.
- DECISIONS는 이미 정한 기술적 선택을 기록한다.
- SELF-DIRECTED OPERATING MANUAL은 Claude가 매 작업을 어떻게 처리할지 설명한다.

따라서 이 문서는 프로젝트의 “직무 매뉴얼”에 해당한다.


======================================================================
1. 영상의 핵심을 어떻게 바꿨는가
======================================================================

영상에서는 Claude Code를 AI 신입사원에 비유하고,
자동화 시스템을 다음 세 요소로 설명한다.

1. Workflow
2. Agent
3. Tool

이를 Music Prompt Architect에 맞게 네 요소로 확장했다.

1. Workflow
- 업무 절차와 완료 기준

2. Agent
- Claude Code

3. Tool
- 파일, Shell, 테스트, Gemini, Skill, Hook, MCP 등

4. Evidence
- 실제로 일이 끝났음을 증명하는 테스트와 결과

Evidence를 추가한 이유는 단순하다.

Claude가 코드를 작성했다는 사실과
그 기능이 제대로 동작한다는 사실은 다르기 때문이다.

운영 문서는 Claude가 항상:

“무엇을 했는가?”뿐 아니라
“어떻게 확인했는가?”까지 남기도록 만든다.


======================================================================
2. 문서를 주면 Claude가 정말 스스로 움직이는가
======================================================================

어느 정도는 그렇다.

Claude Code는 저장소를 읽고,
파일을 편집하고,
명령을 실행하고,
결과를 보고 다음 행동을 선택하는 Agentic Coding Tool이다.

운영 문서를 제공하면 Claude는 다음 판단 기준을 얻게 된다.

- 어떤 문서를 먼저 읽을지
- 현재 Phase를 어떻게 확인할지
- 언제 Plan을 만들지
- 어떤 파일을 탐색할지
- 작업을 얼마나 작게 나눌지
- 어떤 테스트를 실행할지
- 오류가 나면 어떻게 대응할지
- 언제 사용자에게 승인받을지
- 다음 작업을 어떻게 선택할지

그러나 문서 하나만으로 Claude가 완벽하게 강제되는 것은 아니다.

문서는 행동의 기준이다.

반드시 실행돼야 하는 규칙은
Test, Permission, Hook, CI 같은 코드 수준 장치로 강화해야 한다.

예:

“API Key를 커밋하지 마라”
→ CLAUDE.md 규칙

“API Key 패턴이 있으면 Commit을 차단하라”
→ Hook 또는 CI Secret Scan

“변경 후 Type-check를 실행하라”
→ 운영 문서

“Type-check 실패 시 Merge 금지”
→ CI

즉:

문서 = 판단력
Hook/Test/CI = 강제력


======================================================================
3. 왜 CLAUDE.md에 전부 넣지 않았는가
======================================================================

Claude Code는 CLAUDE.md를 프로젝트 규칙으로 읽는다.

하지만 CLAUDE.md가 너무 길고 복잡하면:

- 중요한 규칙이 묻힌다.
- 모든 세션에서 불필요한 Context가 소모된다.
- 서로 다른 절차가 충돌할 수 있다.
- 구체적 Trigger가 없는 긴 지시를 놓치기 쉽다.

따라서 역할을 나눈다.

CLAUDE.md:
- 항상 지켜야 하는 짧은 규칙
- Source of truth 경로
- 핵심 금지 사항

SELF-DIRECTED OPERATING MANUAL:
- 일반 작업 절차
- 계획·실행·검증·기록 루프
- 실패 대응
- 승인 정책

Skill:
- 특정 작업에서 반복되는 상세 절차

Hook:
- 특정 생명주기 시점에 반드시 실행할 동작

이 구조가 Claude가 문서를 최대한 효율적으로 활용하게 한다.


======================================================================
4. 운영 문서 각 장의 의미
======================================================================

------------------------------------------------------------
4.1 프로젝트 운영 아키텍처
------------------------------------------------------------

Claude가 업무를 시작할 때
Workflow, Agent, Tool, Evidence를 구분하게 한다.

이 구분이 없으면 Claude가:

- 문서 작성
- 코드 생성
- 테스트
- 외부 API 호출

을 한 덩어리로 처리하기 쉽다.

구분 후에는:

Workflow가 무엇을 해야 하는지 정하고,
Agent가 판단하고,
Tool이 실행하고,
Evidence가 완료를 증명한다.

------------------------------------------------------------
4.2 지식 우선순위
------------------------------------------------------------

프로젝트에는 여러 문서와 코드가 있다.

서로 충돌할 수 있으므로
Claude에게 무엇을 우선할지 정해준다.

특히:

- 최신 사용자 지시
- DECISIONS
- PRODUCT_SPEC
- 현재 코드와 테스트

의 관계를 명확히 한다.

Claude가 자신의 일반 지식으로
프로젝트 결정을 덮어쓰지 않도록 한다.

------------------------------------------------------------
4.3 세션 시작 프로토콜
------------------------------------------------------------

새 세션마다 Claude가 맥락을 잃지 않게 한다.

Claude는 채팅 기억만 믿지 않고:

- Git status
- 현재 Phase
- 실제 구현
- 테스트
- Source of truth

를 다시 확인한다.

이 덕분에 이미 만든 것을 또 만드는 문제가 줄어든다.

------------------------------------------------------------
4.4 자율 작업 루프
------------------------------------------------------------

운영 문서의 중심이다.

UNDERSTAND
→ INSPECT
→ PLAN
→ EXECUTE
→ VERIFY
→ REVIEW
→ RECORD
→ SELECT NEXT ACTION

이 구조는 Claude가 한 번 코드를 쓰고 끝내지 않고,
결과를 보고 다음 행동을 선택하게 한다.

단, 무한 반복은 아니다.

완료, 승인 대기, 오류 한도, 범위 초과 등
Stop Condition이 있으면 멈춘다.

------------------------------------------------------------
4.5 자기개선 루프
------------------------------------------------------------

영상에서 말한 자기개선 개념을
실제 소프트웨어 개발 방식으로 구체화했다.

예:

Claude가 매번 Provider Profile 필드를 빼먹는다.

나쁜 대응:
- CLAUDE.md에 “앞으로 빼먹지 마” 추가

좋은 대응:
- Zod Schema 보강
- Fixture 추가
- Unit Test 추가
- 반복 절차라면 Skill 갱신

즉 자기개선은
Claude의 다짐이 아니라 시스템의 개선이다.

------------------------------------------------------------
4.6 Skill, Hook, Subagent, MCP
------------------------------------------------------------

Claude Code 확장 기능을 언제 쓸지 설명한다.

Skill:
- 필요할 때 불러오는 반복 업무 매뉴얼

Hook:
- 특정 시점에 반드시 실행하는 자동 검사

Subagent:
- 긴 조사와 로그 분석을 별도 Context에서 처리

MCP/API:
- 외부 시스템을 공식적으로 연결

모든 기능을 처음부터 설치하라는 뜻이 아니다.

반복되는 문제가 생겼을 때
가장 맞는 해결 위치를 선택하기 위한 기준이다.

------------------------------------------------------------
4.7 Context 관리
------------------------------------------------------------

Claude의 효율은 문서 양에 비례하지 않는다.

너무 많은 문서를 매번 읽으면:

- 토큰 소비
- 핵심 규칙 희석
- 오래된 정보 혼입
- 판단 속도 저하

가 발생할 수 있다.

운영 문서는 Claude가 현재 작업에 필요한 정보만 읽게 한다.

긴 내용은 docs와 Skill에 두고,
CLAUDE.md는 가볍게 유지한다.

------------------------------------------------------------
4.8 승인과 Stop Condition
------------------------------------------------------------

“스스로 움직인다”를
“마음대로 모든 것을 바꾼다”로 오해하지 않게 한다.

Claude가 자동 진행할 수 있는 것과
사용자 승인이 필요한 것을 나눈다.

이 프로젝트에서 특히 중요한 승인:

- North Star 변경
- 가사 전면 수정
- DB 파괴적 변경
- 유료 API 대량 호출
- Production 배포
- 인증 변경
- 사용자 데이터 삭제

------------------------------------------------------------
4.9 프로젝트 전용 규칙
------------------------------------------------------------

일반 개발 규칙만으로는
Music Prompt Architect의 핵심을 지킬 수 없다.

따라서 다음을 운영 문서에 넣었다.

- SongDesignSpec 중심
- Provider Prompt는 파생 결과
- 잠긴 가사 보호
- 직설형 가사 존중
- Gemini 구조화 출력
- Compiler/Evaluator 분리
- 국소 수정
- Manual/Copilot/Agent Mode 유지

------------------------------------------------------------
4.10 Task Queue
------------------------------------------------------------

Claude가 다음에 무엇을 할지 스스로 판단하는 방법이다.

우선순위:

P0. 빌드 차단
P1. 보안·데이터 문제
P2. 현재 Phase 필수 기능
P3. 테스트
P4. 문서 불일치
P5. 다음 Phase 준비
P6. 선택 개선

이 우선순위 덕분에 Claude가
디자인 미세 조정을 하다가
빌드 오류를 방치하는 일을 줄인다.

For multi-step work, read:
`docs/CLAUDE_SELF_DIRECTED_OPERATING_MANUAL.md`

Follow its:
- session-start protocol
- understand/inspect/plan/execute/verify/review/record loop
- task-priority rules
- approval gates
- stop conditions

Do not load the full manual for trivial one-file edits.
Do not treat it as permission for destructive, production, or paid actions.

6. Claude에게 처음 입력할 메시지
======================================================================

다음 메시지를 사용할 수 있다.

------------------------------------------------------------

`CLAUDE.md`와
`docs/CLAUDE_SELF_DIRECTED_OPERATING_MANUAL.md`를 읽어라.

현재 저장소, Git status, IMPLEMENTATION_PLAN, DECISIONS,
실제 구현과 테스트를 확인해 현재 Phase를 증거와 함께 판단하라.

현재 Phase의 가장 낮은 미완료 기반부터
우선순위가 있는 Task Queue를 작성하라.

복잡하거나 위험한 변경은 Plan Mode 수준으로 먼저 제안하고,
낮은 위험의 첫 Vertical Slice는
계획 → 구현 → 검증 → 기록 순서로 진행하라.

이미 완료된 기능을 다시 만들지 말고,
현재 범위를 벗어난 개선은 Backlog에 기록하라.

작업 종료 시 Objective, Result, Changed, Evidence,
Known limitations, Next 형식으로 보고하라.

======================================================================
7. 매번 긴 프롬프트를 입력해야 하는가
======================================================================

아니다.

CLAUDE.md가 운영 문서의 경로와 사용 조건을 알려주면,
Claude는 복잡한 작업에서 문서를 읽을 수 있다.

더 자동화하려면 Skill로 만들 수 있다.

예시 위치:

.claude/
  skills/
    project-runner/
      SKILL.md

Skill의 역할:

- 현재 Phase 확인
- Task Queue 작성
- 첫 Vertical Slice 실행
- 검증
- 작업 기록

사용 예:

/project-runner 현재 Phase의 다음 미완료 Slice를 진행해

다만 처음부터 Skill까지 만들 필요는 없다.

먼저 운영 문서가 실제 작업에서 유용한지 확인한 뒤,
반복 사용이 확인되면 Skill로 승격하는 것이 좋다.


======================================================================
8. Hook으로 강화할 수 있는 규칙
======================================================================

운영 문서 적용 후 반복 문제가 보이면 Hook을 추가한다.

추천 후보:

1. Secret scan
- API Key 형태가 파일에 들어가면 차단

2. Format after edit
- 변경 파일 자동 포맷

3. Pre-completion verification
- 완료 전에 Type-check 또는 지정 Test 실행

4. Notification
- Claude가 승인이나 입력을 기다릴 때 알림

5. Dangerous command guard
- 파괴적 명령을 실행 전 차단

모든 규칙을 Hook으로 만들면
속도가 느려지고 관리가 복잡해질 수 있다.

반드시 강제해야 하는 것만 Hook으로 둔다.


======================================================================
9. Subagent는 언제 추가하는가
======================================================================

메인 Claude가 다음 작업 때문에 Context를 많이 잃는다면 추가한다.

- 전체 저장소 구조 조사
- 긴 테스트 로그 분석
- 보안 리뷰
- 공식 API 문서 확인
- 독립 코드 리뷰

예:

repository-explorer:
- 읽기와 검색만 허용
- 저장소 구조와 관련 파일을 요약

test-failure-investigator:
- 테스트 로그 분석
- 원인 후보와 증거 반환
- 코드 수정 권한 없음

security-reviewer:
- Secret, 권한, 입력 검증 검토
- 독립 결과 반환

작곡 Agent와 작사 Agent를 먼저 만들 필요는 없다.

현재는 하나의 MusicDirectorAgent와 Tool Registry가 더 단순하고 안정적이다.


======================================================================
10. 사용자가 Claude에게 어떻게 말하면 좋은가
======================================================================

운영 문서가 있으면 사용자가 파일 단위까지 지시할 필요가 줄어든다.

좋은 입력:

“현재 Phase에서 AgentRun persistence 다음 Slice를 진행해.
기존 Mock E2E를 깨지 말고 실제 Gemini 호출은 아직 하지 마.”

“현재 실패한 테스트를 분석하고,
운영 매뉴얼의 진단 절차에 따라 근본 원인을 수정해.”

“Provider Registry 기능을 구현하되
현재 Phase 범위 안에서 Generic/Suno/Udio만 다뤄.”

사용자가 제공하면 좋은 것:

- 목표
- 범위
- 금지 사항
- 완료 기준
- 특별히 우선할 요소

Claude가 스스로 찾아야 하는 것:

- 관련 파일
- 기존 Pattern
- 테스트 명령
- 현재 Phase
- 필요한 문서
- 작은 구현 순서


======================================================================
11. 문서가 있어도 발생할 수 있는 한계
======================================================================

1. 모호한 요구사항
- Claude가 잘못된 기본값을 선택할 수 있음

2. 오래된 문서
- 실제 코드와 불일치할 수 있음

3. 검증 환경 부족
- API, DB, 브라우저 환경이 없으면 일부 검증 불가

4. 지나치게 큰 작업
- 한 세션에 모든 Phase를 처리하려 할 수 있음

5. 권한 설정
- 자동 편집과 명령 실행 권한이 과도하면 위험

6. 모델 특성
- 같은 문서도 세션과 모델에 따라 세부 행동이 달라질 수 있음

그래서 운영 문서는 다음과 결합해야 한다.

- 명확한 현재 목표
- 작은 Phase
- Test
- Hook
- Git
- 사용자 승인
- 완료 조건

5. 구현

7. 오류 수정
8. 문서 갱신
9. 결과와 다음 작업 보고

즉 사용자는 방향과 승인에 집중하고,
Claude는 프로젝트 운영 절차를 따라 실행한다.


======================================================================
13. 가장 중요한 해석
======================================================================

이 운영 문서의 목적은 Claude에게
더 많은 일을 무조건 맡기는 것이 아니다.

목적은 다음 세 가지다.

1. 사용자가 반복해서 지시하던 업무 절차를 문서화한다.
2. Claude가 다음 행동을 선택할 기준을 제공한다.
3. 그 선택이 틀렸을 때 시스템이 교정될 수 있게 한다.

좋은 자동화는 자율성만 높이는 것이 아니다.

- 목표가 분명하고
- 권한이 제한되고
- 결과가 검증되고
- 변경이 기록되고
- 사람이 통제할 수 있어야 한다.

이것이 영상의 “AI 신입사원” 개념을
실제 프로젝트 운영 문서로 바꾼 핵심이다.
