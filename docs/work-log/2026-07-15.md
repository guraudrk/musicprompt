## 오늘의 요약

오늘부터 커밋·푸시 직전에 개발일지를 자동으로 남기는 절차를 프로젝트에 정식으로 도입했습니다.
사용자가 `docs/docs/CLAUDE_COMMIT_WORKLOG_INSTRUCTIONS.md`라는 지침 문서를 전달하면서, 이걸 읽고
"최초 설정"에 적힌 작업을 그대로 실행하고, 이후부터는 커밋·푸시·PR 생성·작업 마무리 요청이 들어올
때마다 별도 지시 없이 개발일지를 갱신하도록 만들어달라고 요청했습니다. 이번 세션에서는 그 요청에
따라 지침 문서를 정해진 위치로 옮기고, 프로젝트 루트의 `CLAUDE.md`에 영구 규칙을 추가했으며,
`docs/work-log/` 폴더와 `README.md`의 개발일지 색인 섹션을 새로 만들었습니다.

이어서 같은 날 두 번째 세션에서는, 사용자가 "임시방편 말고 최신 AI 기술로 속도와 품질을 동시에
해결하라"고 요구한 것에 답하기 위해 Gemini 컴파일 호출이 실제로 창작 판단이 필요 없는 필드까지
생성하고 있다는 걸 감사(audit)로 찾아내 제거했고(`CompilerOutputSchema` 도입), 그 과정에서 라이브
검증 중 반복적으로 발생한 90초 타임아웃의 원인을 "API 설정이 잘못됐나"라는 사용자 질문에 답하기 위해
직접 진단했습니다. 진단 결과 배열 필드에 상한이 없는 게 원인일 수 있다는 단서를 찾아 실제 스키마에
적용했지만, 적용 후 재검증에서도 타임아웃이 재현돼 완전한 해결로는 확인하지 못했다는 점을 숨기지
않고 그대로 기록했습니다.

## 세션: 컴파일러 스키마 슬리밍 + Gemini 배열 상한 조사

### 요청과 목표

- 사용자가 이전 세션에서 "임시방편적인 프롬프트/재시도 튜닝 말고, 최신 AI 기술을 활용해서 속도와
  이론 반영 품질을 동시에(트레이드오프 없이) 해결할 방법을 찾으라"고 요청.
- 이어서 "일반적인 문장으로 질문했을 때도 우리 작사/작곡 이론이 잘 반영되는지 확인하라"고 요청.
- 라이브 검증 도중 반복적인 타임아웃을 본 사용자가 "혹시 내 Gemini API를 잘 불러오지 못하는 걸까?"라고
  직접 질문 — 이에 대한 명확한 답을 요구.

### 구현 및 변경 내용

- **`src/domain/promptPackage/schema.ts`**: `MusicAIPromptPackageSchema`를 `.omit()`한
  `CompilerOutputSchema`를 새로 추가. `providerId`/`providerDisplayName`/`providerProfileVersion`/
  `profileVerifiedAt`/`strategy`/`theoryRationale`/`warnings`/`toolInstructions`/`promptQuality`/
  `copyBundle` — 10개 필드를 컴파일러(Mock/Gemini)가 더 이상 생성하지 않도록 범위를 좁힘. 또한
  `theoryAddressal`/`unsupportedIntents`/`revisionLevers`/`guidanceTags`/`suggestedProviderIds` 배열
  필드에 `.max()` 상한을 추가.
- **`src/compiler/deterministicFields.ts`**(신규): `buildTheoryRationale`/`buildToolInstructions`/
  `buildWarningsList`/`buildCopyBundle`/`placeholderQuality`/`assembleFullPackage` — 제거된 10개
  필드를 spec/provider/theorySummary로부터 결정론적으로 조립.
- **`src/compiler/pipeline.ts`**: Stage D/G가 `CompilerOutput`을 검증하도록 변경하고, Stage F(평가)
  호출 직전에 `assembleFullPackage`로 전체 패키지를 조립하도록 순서 변경(평가자가 `theoryRationale`
  등 조립된 필드를 읽기 때문).
- **`geminiPromptCompiler.ts`/`mockPromptCompiler.ts`/`compiler/types.ts`/
  `validateTheoryAddressal.ts`**: `CompilerOutputSchema`/`CompilerOutput` 타입으로 교체.
- **`mockOutputBuilders.ts`**: `buildCompilePayload`를 `CompilerOutput` 형태만 반환하도록 단순화하고
  `placeholderQuality`는 `deterministicFields.ts`로 이동.
- **`provider-compiler.system.md`/`prompt-repair.system.md`**: 모델이 생성해야 할 범위가 5개
  창작 필드로 좁혀졌다는 안내 문구 추가.
- **`tests/unit/deterministicFields.test.ts`**(신규, 7개 테스트) 및 `validateTheoryAddressal.test.ts`
  타입 갱신.

### 문제와 해결 과정

- 리팩터링 직후 라이브 검증에서 이전에 빨랐던 문장을 포함해 반복적으로 90초 타임아웃이 발생. 처음에는
  세션 전체의 Gemini 지연으로 의심했으나, 사용자가 "API를 잘 못 불러오는 거 아니냐"고 질문한 것을
  계기로 직접 진단.
- API 키/모델/모드 로딩을 확인(정상). 짧은 지침 + 스키마 없는 최소 호출은 10초 만에 성공 — 설정
  문제가 아님을 확인.
- 앱을 완전히 우회해 실제 시스템 지침(`provider-compiler.system.md`, 6.4KB) + 실제
  `CompilerOutputSchema`로 Gemini SDK를 직접 호출 → 180초를 기다려도 응답 없이 멈추는 현상을 그대로
  재현. 즉 앱 코드 버그가 아니라 이 조합 자체의 문제임을 확인.
- 시스템 지침만 단독(18.6초), 스키마만 단독(10.2초)은 둘 다 빠름 — 조합했을 때만 멈춤을 확인.
- 배열 필드에 `.max()` 상한을 추가한 축소 스키마로 같은 시스템 지침을 테스트하니 16.5초 만에 성공 —
  상한 없는 배열이 원인일 수 있다는 단서를 확보하고 실제 스키마에 적용.
- **적용 후 실제 데모 엔드포인트로 재검증했을 때도 여전히 90초 타임아웃이 발생** — 상한 추가가 확실한
  해결책이라고 확인되지 않음. 이미 상당한 실제 API 시간/비용(수십 분)을 쓴 상태여서, 사용자에게 상황을
  보고하고 추가 라이브 진단을 계속할지 물어봄 → 사용자는 "상한은 유지, 문서화 후 커밋, 추가 라이브
  테스트는 중단"을 선택.

### 기술적 의사결정

- `CompilerOutputSchema`는 `MusicAIPromptPackageSchema`를 `.omit()`으로 파생시켜 두 스키마가 어긋나지
  않도록 함. `SCHEMA_VERSION`은 올리지 않음 — 최종 저장/반환되는 패키지 형태 자체는 그대로이고, LLM이
  생성하는 범위만 내부적으로 바뀐 것이기 때문.
- 배열 상한 값(`theoryAddressal` 7, `unsupportedIntents`/`revisionLevers` 8, `guidanceTags` 8,
  `suggestedProviderIds` 5)은 실제로 이 필드들이 짧다는 Mock의 기존 산출 패턴에 근거해 넉넉하게 설정.
  다만 진단 테스트에서 성공했던 축소 스키마는 상한을 3으로 더 타이트하게 잡았었다는 차이가 있어,
  더 타이트한 값을 다시 시도해볼 여지를 ADR-051에 다음 작업으로 남김.
- 추가 라이브 Gemini 호출을 사용자 결정으로 중단 — 실제 비용이 드는 외부 API를 근거 없이 반복
  호출하지 않는다는 원칙에 따름.

- 일반 문장에 대한 이론 반영 품질(`theoryAddressal`/`structureNotes` 등)을 실제 Gemini 응답으로 직접
  확인하는 작업은, 반복된 타임아웃으로 이번 세션에서 깨끗한 실제 Gemini 응답 샘플을 얻지 못해 완료하지
  못함 — 다음 세션 우선 작업.
- 데모용 spec 생성기(`buildDefaultSpec`+`extractHints`)가 구조/훅/대조 관련 필드를 거의 채우지 않아,
  Mock 폴백 시 `theoryAddressal`/`structureNotes`/`hook`이 비어 보이는 현상을 함께 관찰 — 리팩터링과
  무관한 별도 이슈일 가능성이 있어 다음 세션에서 별도로 조사 필요.

## 세션: 장르별 이론 안내 타겟팅 + Gemini 수요 폭증 발견

### 요청과 목표

- 사용자가 "이론 문서 전체를 통째로 넣고 '이 이론을 적용해서 프롬프트를 만들어'라고 직접 지시하면
  어떨까"라고 제안.
- 이에 대해 크기(38KB vs 현재 6.4KB) 및 ADR-051에서 확인한 위험(긴 지침+스키마 조합이 멈춤을
  유발) 근거로 우려를 전달하자, 사용자가 "전체 대신 입력 문장의 장르/키워드에 맞는 구절만 골라서
  넣자"는 대안을 제시하고 채택.
- "라이브 테스트를 다시 하되 너무 많이 하지 말라"는 조건부 요청.

### 구현 및 변경 내용

- `src/llm/gemini/theoryExcerpts.ts`(신규): `selectGenreTopline(genreTags)` — 이론 문서 §8(장르별
  탑라인)의 6개 장르 안내를 각각 개별 항목으로 분리하고, 입력 spec에 실제 선언된 장르 태그와
  대소문자 무관 매칭. `k-pop`/`kpop`, `ost`/`cinematic` 같은 표기 차이도 인식. 매칭되는 게 없거나
  장르가 아예 없으면 기존처럼 6개 장르 전체 안내로 안전하게 폴백(회귀 방지).
- `provider-compiler.system.md`: 고정 텍스트였던 "Genre topline" 줄을 `{{GENRE_TOPLINE}}` 플레이스
  홀더로 교체.
- `geminiPromptCompiler.ts`의 `compile()`: 호출마다 `input.spec.musicalIdentity.genres`를 읽어
  플레이스홀더를 실제 안내문으로 치환. 기존 유닛 테스트가 `{}`(spec 없음)로 호출하는 경우가 있어
  옵셔널 체이닝으로 방어.
- `tests/unit/theoryExcerpts.test.ts`(신규, 4개): 단일 장르 매칭/다중 장르 중복 제거/미인식 장르
  폴백/무장르 폴백.

### 문제와 해결 과정

- 이 변경을 적용한 뒤 실제 데모 엔드포인트로 딱 한 번(사용자 요청대로 과도하지 않게) 라이브
  검증했는데, 이전과 다른 신호를 얻음: 조용한 90초 타임아웃이 아니라 **서버 로그에 명시적인 Gemini
  에러**가 찍힘 — `500 gemini-3.5-flash is currently experiencing high demand, spikes in demand are
  usually temporary. Please try again later.`
- 이건 우리 코드/스키마 문제가 아니라 Google 쪽에서 해당 모델이 지금 실제로 수요 폭증을 겪고
  있다는 걸 공식적으로 알려주는 응답. 즉 오늘 하루 종일 쫓아다녔던 불안정성(ADR-049~051) 중 상당
  부분이 클라이언트에서 고칠 수 있는 문제가 아니라 외부 용량 문제일 가능성이 새로 확인됨.
- 이번 호출도 결국 Mock으로 폴백되어, 장르 타겟팅 자체가 실제 응답 품질에 미치는 효과는 여전히
  라이브로 확인하지 못함 — 확인된 척하지 않고 그대로 기록.

### 기술적 의사결정

- 사용자가 제안한 "전체 문서 주입"보다 "타겟팅된 발췌"가 크기·관련성 양쪽에서 더 안전한 절충안이라고
  판단하고 그 방향으로 구현. 원래 있던 6개 장르 안내는 폴백으로 보존해 회귀 위험 없음.
- 수요 폭증 에러 발견 이후, 추가 라이브 Gemini 테스트를 사용자 결정에 따라 다시 중단 — 외부 용량
  문제가 원인일 가능성이 있는 상황에서 반복 호출은 근거 없이 비용만 늘림.
